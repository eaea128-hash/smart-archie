# Skill: Security Governance（資安治理）

**觸發時機**：「資安」、「ISO 27001」、「DPIA」、「合規」、「資料保護」、「風險評估」

## 兩個核心框架

### ISO 27001 — 資訊安全管理系統（ISMS）
企業整體資安管理的國際標準，遷移前後都需要驗證。

**10 個控制域（摘要）**
```
A.5  資訊安全政策
A.6  組織的資訊安全
A.8  資產管理
A.9  存取控制            ← 雲端遷移最重要
A.10 加密
A.12 作業安全（Log、備份）← 雲端必查
A.13 通訊安全（網路分割）
A.14 系統取得、開發、維護
A.16 資訊安全事件管理
A.18 法規遵循
```

**雲端遷移 ISO 27001 Checklist**
- [ ] IAM 最小權限原則設定
- [ ] 靜態資料加密（S3/RDS）
- [ ] 傳輸加密（TLS 1.2+）
- [ ] 稽核日誌（CloudTrail / Azure Monitor）
- [ ] 備份與還原測試記錄
- [ ] 網路隔離（VPC / NSG）
- [ ] 廠商資安條款（DPA）簽署

### DPIA（資料保護影響評估）
**觸發條件**：大規模處理個資、高風險資料（醫療/金融/生物識別）

**DPIA 七步驟**
```
1. 描述處理活動   → 資料流圖（Data Flow Diagram）
2. 必要性評估     → 為什麼需要這些資料？最小化原則
3. 識別風險       → 哪些環節可能造成資料外洩、濫用
4. 風險量化       → 可能性 × 影響度 = 風險等級
5. 控制措施       → 技術控制 + 行政控制
6. 諮詢           → 必要時諮詢主管機關（個資委員會）
7. 持續監控       → 每年或重大變更時重做
```

**風險矩陣**
```
影響度 \ 可能性   低      中      高
高               中風險  高風險  極高風險
中               低風險  中風險  高風險
低               可接受  低風險  中風險
```

## 執行步驟

1. **識別資料範圍**：問「這個系統處理哪些個資？誰能存取？」
2. **對照 ISO 27001 控制域**：找出雲端遷移影響的控制項
3. **決定是否需要 DPIA**：有大量個資或高敏感度資料就需要
4. **產出控制措施清單**：技術 + 流程 + 人員培訓
5. **記錄決策**：合規稽核時的書面證明
