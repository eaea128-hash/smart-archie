# Skill: Before Making Changes

**觸發時機**：每次修改功能、重構、或刪除程式碼前

## 執行步驟

1. **確認範圍**：明確說明「我打算修改哪個檔案的哪個函數/區塊」
2. **說明原因**：為什麼需要這個改動？解決什麼問題？
3. **列出風險**：此修改可能影響哪些相關功能？
4. **確認備份**：確認目前 git 狀態乾淨，或提醒 commit 當前狀態
5. **等待確認**：若影響範圍 > 3 個檔案，先列出計畫再詢問使用者是否繼續

## 範例輸出格式

```
📋 修改計畫
- 目標：analyze.html → renderResult()
- 原因：KPI bar 顏色未套用正確 CSS class
- 影響：僅 analyze.html，無後端變動
- 風險：低（純前端 CSS class 修正）
→ 確認繼續？
```
