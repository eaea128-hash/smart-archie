## Step 2｜5 Whys（對每個 A 轉折點追問）

### A1 — Email 跑去錯誤網址

| # | Why | 證據 |
|---|-----|------|
| W1 | 為什麼驗證信指向舊網址？ | URL 含 `ai-product-analyzer.netlify.app` |
| W2 | 為什麼 Supabase 用舊 URL？ | Supabase Dashboard → Site URL 從未更新 |
| W3 | 為什麼 Site URL 沒有更新？ | 開發時從舊專案複製 Supabase 設定 |
| W4 | 為什麼沒有 checklist 提醒？ | 部署前沒有「環境設定核對清單」 |
| W5 | 為什麼沒有這份清單？ | 沒有建立「新站上線 SOP」的習慣 |

**根本動作**：新站部署 checklist 中必須含「Supabase Site URL 核對」

---

### A2 — 406 大量出現

| # | Why | 證據 |
|---|-----|------|
| W1 | 為什麼 quota_usage 查詢 406？ | PostgREST `.single()` 在 0 筆時回傳 PGRST116（406）|
| W2 | 為什麼用 `.single()` 而非 `.maybeSingle()`？ | 開發時假設「用戶一定有 quota 記錄」 |
| W3 | 為什麼這個假設是錯的？ | 新用戶首次登入時 quota_usage 沒有當月記錄 |
| W4 | 為什麼沒有在 code review 抓到？ | 沒有針對「新用戶首次登入」的測試案例 |
| W5 | 為什麼沒有這個測試案例？ | 缺乏 E2E 新用戶 onboarding flow 測試 |

**根本動作**：所有 Supabase 單筆查詢一律用 `.maybeSingle()`，加入 lint rule

---

### A3 — 500 analyses 欄位不存在

| # | Why | 證據 |
|---|-----|------|
| W1 | 為什麼 `project_name` 欄位不存在？ | 錯誤訊息：`column analyses.project_name does not exist` |
| W2 | 為什麼 DB schema 和程式碼不一致？ | 程式碼功能迭代後沒有同步更新 DB |
| W3 | 為什麼沒有同步？ | Schema migration 是手動執行，容易遺漏 |
| W4 | 為什麼是手動？ | 沒有採用 Supabase migrations CLI 流程 |
| W5 | 為什麼沒採用？ | 早期快速開發優先，跳過 migration 管理 |

**根本動作**：建立 `supabase/migrations/` 目錄，所有 schema 變更走 migration 檔案

---

### A4 — Netlify Credits 超限

| # | Why | 證據 |
|---|-----|------|
| W1 | 為什麼 Credits 超限？ | 52 次 production deploy × 15 credits = 780（超過 500 限制）|
| W2 | 為什麼有 52 次 deploy？ | 每個小 bug 修完就立刻 push |
| W3 | 為什麼每次都 push？ | 沒有「批次合併後再 push」的紀律 |
| W4 | 為什麼沒有這個紀律？ | 沒有明確的 commit 策略規範 |
| W5 | 為什麼沒規範？ | 缺乏 session 開始前的「工作方式協議」 |

**根本動作**：每個 session 開始時 Claude 主動宣告「本 session 批次 push 策略」

---

### A5 — RAG 403 全部失敗

| # | Why | 證據 |
|---|-----|------|
| W1 | 為什麼 rag-ingest 回 403？ | `需要管理員權限`，getAdminUser() 回傳 null |
| W2 | 為什麼 getAdminUser 回傳 null？ | profiles 表找不到該 user 的 role='admin' |
| W3 | 為什麼 profiles 表沒有資料？ | Supabase trigger 沒有在 signUp 後自動建立 profile |
| W4 | 為什麼 trigger 沒有執行？ | Supabase Schema 中的 trigger 可能沒有在 SQL Editor 執行 |
| W5 | 為什麼沒有驗證 trigger 有無生效？ | 部署後沒有「新用戶註冊完整流程」測試 |

**根本動作**：`supabase-schema.sql` 執行後必須用測試帳號走完完整 registration → login → dashboard 流程

## Step 3｜根因（Root Cause）

從以下標準清單選出**唯一主要根因**：

- [ ] 邏輯錯誤（Algorithm）
- [ ] 邊界條件未處理（Edge Case）
- [x] **環境/設定不一致（Config Drift）** ← 主要根因
- [ ] 依賴版本衝突（Dependency）
- [ ] 測試覆蓋不足（Test Gap）
- [ ] 需求理解錯誤（Misunderstanding）

**理由**：本 session 80% 的問題源自「程式碼期待的環境設定」與「實際環境」之間的落差：
- Supabase Site URL 沒更新
- DB schema 欄位比程式碼少
- profiles 表沒有 trigger 建立資料
- Netlify 環境變數名稱打錯（PENAI_API_KEY）

所有這些都是 **Config Drift** — 程式碼跑在一個「它以為是 X 但實際是 Y」的環境。

## Step 4｜反事實（Counterfactual）

> 如果使用者完全沒有介入（沒有截圖、沒有指出錯誤），我會在哪裡卡住？

| 轉折點 | 若無介入的結果 |
|--------|--------------|
| A1 Email | 認為 email 功能正常，繼續開發其他功能，上線後用戶全部收到壞連結 |
| A2 406 | 判定「Supabase 查詢成功但回 null」是正常行為（因為有 `?.` 防護），不會追查 406 |
| A3 500 | fallback 程式碼遮蔽了真正錯誤（column missing），認為功能正常，dashboard 永遠空白 |
| A4 Credits | 繼續 push 直到完全無法部署才發現 |
| A5 RAG 403 | 誤判為「token 格式問題」或「rate limit」，反覆嘗試不同格式 |

**最危險的盲點**：A3 — fallback 機制把真實錯誤藏起來，讓問題看似「優雅降級」，實際上是靜默失敗。

## Step 5｜觸發-行動清單（Trigger-Action List）

> 格式：When {觸發}, before {階段}, must {行動}

```
When starting a new debug session,
before writing any code,
must declare batch-push policy: "all changes this session → single push at end"

When querying Supabase for a single row,
before writing .select().eq().single(),
must use .maybeSingle() instead and handle null case

When deploying to a new Netlify site,
before first user registration,
must verify Supabase Site URL and Redirect URLs in Auth > URL Configuration

When adding new columns to DB schema,
before updating application code,
must create a migration file in supabase/migrations/ with timestamp prefix

When implementing admin-only API endpoints,
before going to production,
must test with: (a) no profile row, (b) profile with role≠admin, (c) valid admin

When a Netlify Function returns 500 with Content-Type: text/plain,
before assuming it's a business logic error,
must check if the crash happens at module initialization (top-level code)

When user registers successfully,
before granting dashboard access,
must verify profiles table row was created by checking auth.users JOIN profiles

When setting OPENAI_API_KEY in Netlify,
before triggering any deployment,
must double-check key name spelling (not PENAI_API_KEY)

When FROM email address uses a custom domain,
before sending any transactional email,
must verify domain is added and verified in Resend dashboard

When RAG ingest returns 403,
before assuming token issue,
must verify: (a) profiles row exists, (b) role='admin', (c) token is fresh
```

